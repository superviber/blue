[PERSPECTIVE P01: Inline writeback with a read-only convention, not a separate mapping file]
Scone argues writeback should go to `.sync/jira-keys.yaml` to keep authored artifacts pristine. I understand the coupling concern, but I maintain that inline is correct -- with a refinement. The onboarding signal of `jira_url` in front matter is not a nice-to-have; it is the single most important indicator to a new contributor that a file is live. A separate mapping file means a new teammate opens a story file, sees no trace of Jira, and has to know to look in `.sync/` -- that is a hidden knowledge dependency that defeats the purpose of a self-documenting PM repo. However, I concede Donut's point about merge conflicts. The resolution: writeback fields should be placed in a clearly delimited block at the bottom of front matter (after a `# sync` comment marker), and tooling should treat that block as machine-owned. `git merge` drivers or `.gitattributes` can mark those lines as "ours" on conflict. This gives you inline visibility AND conflict isolation.

[REFINEMENT: T01 -- inline writeback with machine-owned block convention]
Instead of choosing between inline and mapping file, define a convention: sync-generated fields (`jira_url`, `jira_key`) live in a `# --- blue-sync ---` delimited section at the end of YAML front matter. This section is machine-owned -- humans read it but never edit it. Tooling (`blue sync`) overwrites it idempotently. Merge conflicts in this section are auto-resolved by taking the latest sync value. This preserves the onboarding DX I advocated (the URL is right there in the file) while addressing Scone's pristine-artifact concern (the authored fields above the marker are untouched by sync) and Donut's merge-conflict concern (the block is small, isolated, and auto-resolvable).

[PERSPECTIVE P02: First-sync safety needs a three-state model, not just --confirm]
My T04 from Round 0 proposed `--confirm` on first sync. After reading Croissant's point about Jira-habituated teammates and Muffin's DocSource trait direction, I want to sharpen this. The first-sync experience should have three states: (1) `blue sync --dry-run` which is the default on first detection and shows exactly what would be created, with counts and a preview table; (2) `blue sync --confirm` which executes after the user has reviewed the dry-run; (3) subsequent `blue sync` which runs implicitly because writeback markers already exist. The key insight: the presence of any `jira_url` in any file is itself the signal that this repo has been through first-sync. No additional state file needed -- the writeback IS the state. This ties back to T01: if writeback goes to a mapping file, you lose this elegant detection.

[TENSION T04: First-sync state detection depends on T01 resolution]
If T01 resolves toward a mapping file (Scone's position), first-sync detection requires checking `.sync/jira-keys.yaml` existence -- a file that a contributor might not know about or might accidentally delete. If T01 resolves toward inline (my position), first-sync detection is simply "does any file have a `jira_url` field?" which is self-evident and robust. The two tensions are coupled: T04's ergonomics improve when T01 resolves toward inline.

[PERSPECTIVE P03: Collaborative editing friction is real but solvable with a status shortcut]
Croissant's T07 is the deepest human-factors tension in this dialogue. Git PRs for status updates are absurdly high-friction compared to dragging a Jira card. But full bidirectional sync is a complexity cliff. The middle path: `blue status <file> <new-status>` as a CLI command that edits the front matter status field and auto-commits with a conventional prefix. This gives a Jira-habituated teammate a one-liner instead of a PR workflow. For non-CLI users, pair this with Croissant's drift detection: if someone updates status in Jira, `blue sync --check-drift` flags it, and the next sync can optionally accept Jira-side status changes as authoritative for the `status` field only. This is not full bidirectional sync -- it is a narrow, field-scoped read-back that acknowledges the reality that status is the one field people will always update in Jira first.

[CONCESSION: T07 -- status field deserves special treatment]
Croissant is right that the system quietly optimizes for solo-founder control. I concede that the `status` field (and only status) should support Jira-to-git read-back as a first-class feature, not just drift detection. The rationale: status is the highest-frequency update, the lowest-risk field to accept from Jira (it does not affect structure or dependencies), and the field that Jira-habituated teammates will refuse to update via git. Making status bidirectional while keeping everything else unidirectional is a pragmatic compromise that preserves git authority over structure while acknowledging Jira authority over workflow state.
