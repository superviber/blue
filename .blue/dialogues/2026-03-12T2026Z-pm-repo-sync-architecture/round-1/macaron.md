[PERSPECTIVE P03: "Project then verify" concretely means a post-sync read-back that produces a DriftReport, not warnings on stdout]
Muffin's Round 1 refinement of T05 proposes `blue sync --verify` emitting warnings. I agree with the separation from the sync engine, but warnings alone are insufficient for PM workflows. The read-back must produce a structured `DriftReport` artifact (JSON or YAML) that captures: (a) Jira-side status changes not reflected in git (e.g., story moved to "Done" in Jira board but still `status: in-progress` in markdown), (b) sprint lifecycle mismatches (sprint closed in Jira but stories still assigned to it in git), (c) field-level deltas (points changed, assignee changed). This report is the input to three consumers: CI gates (`blue sync --check` fails the build on category-a drift), dashboard rendering (show drift badges), and the human decision of whether to accept Jira-side changes back into git. The DriftDetector component Muffin proposes is right, but its output type matters as much as its existence -- `Vec<DriftWarning>` should be `DriftReport { items: Vec<DriftItem>, sprint_mismatches: Vec<SprintDrift>, summary: DriftSummary }` so consumers can filter by severity.

[PERSPECTIVE P04: depends_on must carry link-type semantics to avoid lossy projection]
Cupcake raised T03 correctly: Jira links are typed (blocks/is-blocked-by/relates-to) while PM YAML `depends_on` is a flat list. From a PM workflow perspective, "depends_on" implicitly means "blocks" -- story A cannot start until story B completes. But not all cross-story relationships are blocking. The PM schema should support `depends_on` (maps to Jira "is blocked by"), `relates_to` (maps to Jira "relates to"), and optionally `blocks` (explicit forward declaration). Without this, the two-pass link creation will project every relationship as the same Jira link type, which misrepresents the dependency graph and makes Jira-side sprint planning unreliable. This is not schema bloat -- it is the minimum semantic fidelity needed for the DAG validation I proposed in Round 0 (P02) to produce meaningful results. A `depends_on` cycle is a hard error; a `relates_to` cycle is fine.

[PERSPECTIVE P05: T01 resolution needs a writeback-triggers-huddle convention for crew workflows]
I accept Muffin's inline `jira_url` single-field writeback as the right resolution for T01 -- Eclair's onboarding argument is compelling and `.gitattributes` merge driver handles the conflict concern. But from a PM workflow perspective, writeback creates a new coordination event in the crew model. When a meeple runs `blue sync` and gets writeback commits, other meeples need those `jira_url` values to correlate their work with Jira. This means: (1) `blue sync` with writeback should auto-deliver the writeback commit (not wait for manual `/lc-crew deliver`), and (2) other meeples should huddle before their own sync to pick up existing `jira_url` mappings and avoid duplicate Jira issue creation. The convention: "sync implies deliver of writeback commits; cook implies huddle before sync."

[RESOLVED T01]
Inline `jira_url` writeback with `.gitattributes` merge driver, per Muffin's proposal. Adding the convention that writeback commits auto-deliver to prevent duplicate issue creation across meeples.

[REFINEMENT: T05 from "unidirectional vs read-back" to "unidirectional projection + structured drift reporting"]
The sync engine stays strictly git-to-Jira. Read-back is a separate `DriftDetector` component (agreeing with Muffin) that produces a typed `DriftReport`, not just warnings. Three operational modes: `--verify` (print report), `--check` (CI gate, nonzero exit on blocking drift), `--check-drift` (Croissant's always-on CI variant). Sprint lifecycle drift (closed sprint, changed velocity) is a distinct category from field-level drift and should be surfaced separately because it requires different remediation (you update the sprint file, not the story file).

[TENSION T03: depends_on needs link-type semantics or the DAG is lossy]
Flat `depends_on` lists cannot distinguish blocking from non-blocking relationships. Projecting all relationships as a single Jira link type misrepresents the dependency graph and undermines sprint-ordering validation. The PM schema needs at minimum `depends_on` (blocking) and `relates_to` (non-blocking) as separate front matter fields.
