[PERSPECTIVE P03: T01 resolution -- hybrid writeback with commit-convention separation]
The T01 tension (inline front matter vs mapping file) has a clean resolution once you accept that the two sides are optimizing for different audiences at different times. Inline `jira_url` in front matter serves the *reading* path -- a contributor opens a file and immediately sees its Jira context (Eclair's onboarding argument). The mapping file (`.sync/jira-keys.yaml`) serves the *writing* path -- sync commits touch one file instead of N, eliminating merge conflicts (Scone's coupling concern, my R0 conflict concern). The resolution: do both. `blue sync` writes to `.sync/jira-keys.yaml` atomically in a single commit, then a post-sync step injects `jira_url` into front matter as a *derived* field. The key insight is commit convention: mapping-file commits use `blue-sync: map` prefix and are auto-generated; front-matter injection commits use `blue-sync: annotate` prefix and are optional (controlled by `writeback_policy` in jira.toml per Cupcake's suggestion). If a merge conflict occurs on the annotation commit, it can be safely dropped and regenerated -- the mapping file is the source of truth for the sync relationship, the front-matter annotation is a convenience projection of it. This gives Eclair the DX win, Scone the clean separation, and eliminates the "which one is authoritative" ambiguity entirely.

[PERSPECTIVE P04: T02 resolution -- branch-based serialization with worktree-aware locking]
My R0 tension (T02, concurrency) interacts directly with the meeple/worktree model. Multiple meeples share a single PM repo (or each have their own worktree of it). Writeback must be serialized not at the file level (too granular, deadlock-prone) but at the *sync operation* level. The mechanism: `blue sync` acquires an advisory lock by pushing a lightweight lock-ref (`refs/sync/lock`) to the PM repo's remote before starting writeback. If the ref exists and is less than 5 minutes old, sync waits or fails with a retry hint. After writeback commit + push, the lock-ref is deleted. This is git-native (no external coordination service), works across worktrees and machines, and degrades gracefully (stale locks auto-expire). For the common solo-founder case, the lock is never contended. For multi-meeple crews, it serializes syncs at roughly the cadence they would naturally occur. The 5-minute TTL prevents deadlocks from crashed sessions. Combined with P03's mapping-file approach, the lock window is short because writeback touches one file, not dozens.

[PERSPECTIVE P05: Cross-repo coordination must flow through the PM repo's git history, not sidecar state]
Strengthening my R0 P02: several agents proposed `.blue/jira/last-sync.yaml` or similar sidecar state in code repos to track sync status. This is the wrong rendezvous point. The PM repo's commit log is already a complete audit trail of every sync operation (via the `blue-sync:` commit prefix convention). Any code repo that needs to know "has story X been synced?" should query the PM repo's mapping file, not maintain its own shadow. The `blue_uuid` in story files is the join key; the PM repo's HEAD is the sync state. Adding per-code-repo sync state creates N+1 sources of truth where N is the number of code repos in the org. The org layout (RFC 0067) already provides the mechanism to locate the PM repo from any code repo via `domain.yaml` -- use that path, do not duplicate it.

[REFINEMENT: T01 -- proposing RESOLVED via hybrid model]
T01 can be marked resolved if the panel agrees on: (1) `.sync/jira-keys.yaml` is the authoritative mapping, committed atomically by sync; (2) front-matter `jira_url` is a derived annotation, optional per `writeback_policy`, regenerable from the mapping file; (3) commit conventions distinguish the two (`blue-sync: map` vs `blue-sync: annotate`). This satisfies Scone (pristine authored artifacts), Eclair (onboarding DX), and eliminates the "which is source of truth" ambiguity.

[REFINEMENT: T02 -- proposing RESOLVED via lock-ref mechanism]
T02 can be marked resolved if the panel agrees on: (1) `refs/sync/lock` advisory lock on PM repo remote, acquired before writeback, released after push; (2) 5-minute TTL for stale lock expiry; (3) combined with single-file mapping writeback (P03), lock window is minimal. This addresses Muffin's atomicity concern (mapping file is one atomic commit) and my R0 concurrency concern (serialized via lock-ref).

[TENSION T09: Lock-ref mechanism assumes push access to PM repo remote]
The `refs/sync/lock` approach requires every sync actor to have push access to the PM repo. In an org where PM repo write access is restricted (e.g., only tech leads can push), meeples in code-repo worktrees cannot acquire the lock. This creates an implicit access-control coupling between sync permissions and PM repo write permissions that needs to be addressed -- possibly by having `blue sync` in code repos delegate writeback to a PM-repo meeple via a queue, or by requiring that sync always runs from a PM repo worktree.
