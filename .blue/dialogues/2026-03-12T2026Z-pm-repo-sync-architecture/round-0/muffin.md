[PERSPECTIVE P01: Front matter schema should be the abstraction boundary, not repo mode]
The notes frame the problem as two "modes" (Blue repo vs PM repo), but the real axis of variation is the document schema: what fields exist, how they parse, and what they map to in the tracker. Introduce a `DocSchema` trait (or enum) that describes field extraction, writeback format, and type mapping -- then repo detection just selects the schema. This keeps the `IssueTracker` trait untouched and avoids a proliferating mode switch in the sync engine. The current `JiraBinding` struct with `blue_uuid`/`task_key`/`epic_id` is already schema-specific to RFCs; PM docs need `id`, `epic`, `status`, `points`, `sprint`, `assignee`, `depends_on`, `labels` -- a fundamentally wider surface that cannot be shoehorned into the same binding type.

[PERSPECTIVE P02: depends_on demands a two-pass sync with deferred linking]
The `depends_on` field references other PM doc IDs (e.g., `VOT-002`) that may not yet have Jira keys at sync time. The sync engine must first create all issues (collecting the id-to-jira-key mapping), then make a second pass to create issue links. This is a sequencing constraint absent from the current single-pass RFC sync and should be designed into the `DocSchema` contract explicitly rather than bolted on.

[TENSION T01: Writeback granularity vs atomicity]
Writing `jira_key` back into YAML front matter after creation means `blue sync` mutates the working tree. For RFCs this was a single field in a table row; for PM repos it touches dozens of files across epics/. If sync is interrupted mid-writeback, the repo is in a partially-written state with some docs having keys and others not. The notes do not address whether writeback should be atomic (batch commit) or incremental, and this choice affects idempotency guarantees.
